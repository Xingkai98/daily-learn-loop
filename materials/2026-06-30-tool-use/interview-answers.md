# Tool Use / Function Calling — 模拟面试参考答案

---

## 问题 1：Tool Use 完整生命周期

**评分要点：**
- 能画出完整的状态流转图
- 能区分不同超时场景的处理策略
- 理解模型如何"知道该停"

**满分回答示范：**

完整生命周期如下：

**Phase 1：意图识别。** 模型接收 user message + system prompt（含 tool definitions）。在生成 token 的过程中，模型在"正常文本"和"tool call"两种输出模式间切换。切换依据是微调训练中学会的模式——模型在训练数据中见过大量 tool call 样本，学会了在需要外部信息时输出特定的结构化 token。

**Phase 2：结构化输出。** 模型输出 tool call（工具名 + JSON 参数）。以 OpenAI 为例，输出的是 `tool_calls` 数组，包含 `id`、`function.name`、`function.arguments`。这步是"普通 token 输出"的一部分，但解析器会识别特殊格式并拦截。

**Phase 3：工具执行。** 框架解析 tool call，调用对应的 Python/JS 函数。这一步是纯工程——不是模型在跑。

**Phase 4：结果注入。** 工具返回值作为一条新消息（role: "tool"）追加到对话历史末尾。模型基于包含工具结果的新上下文继续生成。

**Phase 5：最终输出。** 模型综合工具结果，生成给用户的自然语言回复。

**异常分支：**
- **超时**：5 秒超时 → 快速重试（可能是网络抖动）；30 秒超时 → 返回降级响应（"该服务暂时不可用"），不给用户留白等待。
- **解析失败**：模型输出的 JSON 格式不对 → 用 error message 喂回模型让其重试（self-correction）。
- **工具不存在**：schema mismatch → 返回明确的错误信息，不重试（重试也不会变对）。
- **权限不足**：Auth 403 → 告知模型"权限不足"，模型转化为用户友好的"抱歉，我没有权限执行这个操作"。

**关于并行判断：**
如果模型在一次推理中输出了多个 tool_call，框架检查它们之间是否有参数依赖。没有依赖 → 并行执行（Promise.all）；有依赖 → 按顺序执行。目前大多数框架按"同一轮输出"来判断——同一轮输出的 tool call 之间被认为独立。

**常见错误：**
- 说工具是"模型内部执行的"——工具在模型外部，由框架执行
- 不讲异常分支——happy path only，缺乏工程经验
- 把 tool call 和 chain-of-thought 混淆——CoT 是内部推理，tool call 是外部交互

## 问题 2：Schema 设计与语义理解

**评分要点：**
- 理解 description 是模型的"说明书"
- 能设计有区分度的 schema
- 有处理大规模返回结果的策略

**满分回答示范：**

**schema 设计：**

```yaml
search_users:
  description: >
    模糊搜索用户。当用户提供的查找条件不精确时使用此工具——
    比如"找一下张三"、"有没有姓李的人"、"技术部的同事"。
    返回匹配的用户列表（最多 20 条），包含 id、姓名、部门。
    如果找到了唯一匹配，下一步请用 get_user 查询详情。
    注意：此工具只做搜索，不会返回用户的完整信息。
  parameters:
    query: 搜索关键词（姓名、部门、职位等）
    limit: 返回数量上限，默认 10，最大 50

get_user:
  description: >
    精确获取单个用户的完整信息。当你已经知道用户的 ID 时使用。
    不要用此工具做搜索——如果没有确切 ID，请先用 search_users。
    返回完整 profile：联系方式、入职日期、汇报关系等。
  parameters:
    user_id: 用户唯一 ID（由 search_users 返回）
```

**为什么这样设计：**
关键是在 description 中建立"工作流依赖"——search_users 的 description 明确告诉模型"拿到 ID 后用 get_user"，而 get_user 的 description 也反过来告诉模型"没有 ID 就先去 search_users"。这种双向引导大幅降低选错工具的概率。

**"帮我找一下张三"的处理：**
search_users(query="张三")——因为条件是模糊的（"找一下"），没有提供 ID。

**如果模型选错了：**
改 description。通常是因为两个工具的 description 没有明确的"什么时候用我/什么时候不用我"的边界，需要加入排他性描述。

**50 个结果的处理：**
不要直接把 50 个 JSON 塞给模型。这浪费 token 且模型处理大量结构数据的效果不好。策略是：
- 只取前 5-10 个最相关的给模型
- 或者让模型向用户追问以缩小范围（"我们找到了 50 个匹配的人，你能提供更多信息吗？比如部门或入职时间？"）
- 如果模型判定有唯一匹配，直接取第一个查详情

**量化指标：**
- **工具选择准确率**：模型选了正确工具的占比（需要人工标注 ground truth）
- **参数填充 F1**：必填参数的填充完整率和准确率
- **不必要的调用率**：该直接回答的问题却调了工具的比例
- **端到端成功率**：用户问题得到满意回复的比例（这是最终目标）

**常见错误：**
- description 太短太笼统——"用于搜索用户"确实不够
- 没说清楚工具间的调用顺序依赖
- 没有考虑返回结果过多的情况

## 问题 3：可靠性工程

**评分要点：**
- 有分层容错的思路
- 能区分可重试和不可重试的错误
- 有生产监控的经验

**满分回答示范：**

**分层容错体系：**

**L1 — 客户端校验：**
- 在发 tool call 之前验证参数格式（不合规的直接改，不改的就返回给模型重试）
- Schema 级别的类型校验

**L2 — 重试策略：**
- 可重试：网络超时、503、限流（429）→ 指数退避（1s → 2s → 4s），最多 3 次
- 不可重试：参数错误（400）、权限不足（403）、资源不存在（404）→ 直接返回 error message 给模型，让模型决定下一步
- 业务错误（"库存不足"、"余额不够"）：不是系统故障，直接告知用户

**L3 — Fallback / 降级：**
重试耗尽后的策略：
- 有替代工具就用替代的（比如查不到物流就去查订单状态）
- 没有替代就走"优雅降级"——模型向用户回复："抱歉，XX 服务暂时不可用。我已经记录了您的问题，会在服务恢复后第一时间通知您。"
- 绝对不要直接给用户看 stack trace 或原始 JSON error。

**L4 — Schema 漂移防御：**
工具 API 的响应格式可能变更。防御策略：
- 在解析工具返回值时做 loose parse——用 schema 的 subset 匹配，允许未知字段
- 对关键字段做类型检查，非关键字段缺失不影响流程
- 如果返回值完全无法解析，当作"工具异常"，走 L3 fallback

**监控指标：**
- **调用成功率** = 成功调用 / 总调用（建议分工具看，不只看总量）
- **首调成功率**（first-attempt success rate）：第一次就成功的比例，衡量模型生成 tool call 的质量
- **重试率** = 需要重试的调用 / 总调用
- **端到端延迟** P50/P95/P99（按工具分）
- **错误分布**：按错误类型（超时/参数错/权限/业务）的占比
- **用户满意度**：调用后的用户反馈——用户点了"踩"还是继续聊

**常见错误：**
- 所有错误都重试——业务错误重试无意义
- 没有降级路径——fallback 就是"报错"
- 不讲监控——缺乏生产意识

## 问题 4：Multi-Turn & 上下文管理

**评分要点：**
- 理解上下文窗口的压力来源
- 有工具结果压缩/摘要的策略
- 能区分"需要保留"和"可以丢弃"的信息

**满分回答示范：**

**问题分析：**
每轮对话包含：user message + assistant message（可能含 tool_calls）+ tool_response。tool_response 通常是一个完整的 JSON，可能很大。6 次工具调用可能占了 60% 的上下文。

**策略：**

**只保留必要信息：**
- tool_call（模型输出的调用指令）：**保留**，因为模型需要知道"我调了什么"来维护 reasoning 链
- tool_response（原始 JSON）：**可以压缩**。模型只需要关心的不是原始 JSON 结构，而是其中的关键信息字段
- 模型对 tool_response 的总结（给用户的那段话）：**保留**，这是模型理解上下文的关键

**工具结果摘要器：**
对每次 tool_response 做结构化提取：
- 成功 → 只保留关键字段（如 `{"order_id": 123, "status": "shipped", "eta": "2026-07-03"}`），丢弃元数据和冗余字段
- 失败 → 保留错误类型和关键 message，丢弃 stack trace

**风险：**
摘要可能丢失关键信息——比如一个看似无关的字段在下一轮变得重要。缓解：对摘要后的上下文做 key-value 索引，当后续 tool call 引用到被丢弃的信息时可以回溯。

**判断工具结果是否还需要：**
- 如果是一次性查询（"今天的天气"），下轮对话就可以丢弃原始结果
- 如果是会频繁引用的数据（"这个订单的所有信息"），保留摘要

**与 KV Cache 的关系：**
KV Cache 是模型推理层的优化——缓存了 attention 的 K 和 V，避免每步重算。但 KV Cache 的大小正比于上下文长度。如果你能减少上下文中工具结果的 token 数，KV Cache 也会等比缩小。两者是正相关的——上下文管理做好了，KV Cache 压力也小。

**常见错误：**
- 说"全保留"——不考虑成本
- 说"全丢掉"——不考虑后续引用
- 不讲摘要的风险——过度乐观

## 问题 5：安全与权限

**评分要点：**
- 理解认证和授权的分层
- 能识别 prompt injection 对 tool use 的威胁
- 有 Human-in-the-Loop 设计思路

**满分回答示范：**

**权限模型：**

Tool call 携带的是**用户身份**，不是 Agent 身份。Agent 只是一个代理——用户说"删掉我的订单"，Agent 调用 `cancel_order(order_id)`，权限校验应该检查**当前用户是否有权取消这个订单**。

三层校验：
1. **认证层**（AuthN）：验证调用者是谁（JWT/session token 传递）
2. **授权层**（AuthZ）：验证该用户是否有权执行该操作（RBAC / OPA）
3. **业务层**：验证数据归属（这个订单是否属于该用户）

**delete_user 的安全设计：**
- 如果 `delete_user` 没有 user_id 参数（隐式使用当前登录用户），风险是：如果 Agent 被 prompt injection 诱导调用 delete_user，当前用户就被删了
- 安全设计：所有变更类操作强制传 `user_id`，后端校验 `user_id == current_user_id`，不允许隐式代理

**Prompt Injection 防护：**
- **输入过滤**：用户输入中的 `---SYSTEM---`、tool_call 关键字增加权重标记
- **工具白名单**：高风险操作（delete、transfer）不在默认工具列表中，需要额外确认
- **独立校验层**：即使模型输出了 tool call，也在执行前由独立的规则引擎校验——"模型说删谁、用户是本人吗、操作目标属于用户吗"

**Human-in-the-Loop 设计：**
分层确认：
- L0（自动通过）：只读操作 + 查询本人数据
- L1（软确认）：修改本人数据 → 在 UI 上显示即将执行的操作，用户点击确认
- L2（硬确认）：删除、支付、权限变更 → 需要用户输入验证码或二次认证
- L3（人工审核）：金额 > 阈值、批量操作、异常模式 → 进入人工审核队列

**常见错误：**
- 把认证和授权混为一谈
- 忽略了 prompt injection 的攻击面
- 所有操作都需要确认——影响用户体验，应该分层

## 问题 6：Agent 自主性与编排

**评分要点：**
- 理解目前 LLM 工具编排的真实局限
- 能举出实际的失败案例
- 有工程兜底的设计思路

**满分回答示范：**

**当前局限：**

LLM 的工具编排本质上是"每步做局部最优决策"。它没有全局规划能力——不能像 A* 搜索那样从目标反推需要什么工具。表现出来就是：
- 短视：选了当前看起来最好的工具，但几步后走入死胡同
- 不回溯：发现走错了不会自动回到分支点
- 循环：用不同参数调用同一个工具，期望结果改变

**实际翻车的例子：**
用户：**"我上周在你们网站买了三样东西，其中有一个还没到，帮我看一下。"** 
Agent 的翻车路径：调用 search_orders → 返回 3 个订单 → 随机挑了一个查物流 → 发现已签收 → 认为任务完成，告诉用户"您的订单都已签收"——但实际上查的是错的那个订单。

应该是：查到 3 个订单 → 逐一查物流 → 发现其中 1 个未签收 → 汇报具体是哪个没到。

**ReAct vs 纯 Tool Use：**
- 纯 Tool Use：模型直接输出 tool call，没有显式的中间推理
- ReAct（Reasoning + Acting）：模型先输出一段推理（"我需要先..."），再输出 tool call。推理文字帮助模型厘清思路，减少短视问题。ReAct 更适合需要多步推理和分支判断的复杂场景。

**防止循环调用：**
工程上做三层防护：
1. **重复检测**：连续 2 次调用同一个工具 + 同一个参数 → 阻止第三次，强制模型输出自然语言，向用户求助
2. **语义循环检测**：即使参数不同，如果连续 N 次调用同类工具且每次都"没找到"，视为循环
3. **Max turns**：硬限制单次对话最多 10 轮 tool call，超了就终止并转人工

**3 轮判断法：**
如果 Agent 3 轮 tool call 后没有任何实质进展（没有获取到新信息、没有缩小问题范围、没有改变行动方向），大概率是在瞎转。实质进展的标准是每一轮 tool_response 的信息量在递增或问题空间在缩小。

**长期趋势：**
我认为工具编排应该**模型能力 + 工程约束两手抓**：
- 模型侧：更强的 planning（如 o1-style 的思考链 + 工具调用），更好的全局规划
- 工程侧：状态机 + Guardrails（对于关键流程，不应该依赖模型的自由决策，而应该限定在预定义的流程图中走）

两者不是对立的——模型能力越强，工程约束可以越松；但永远需要工程兜底。

**常见错误：**
- 高估模型能力——认为"模型能自己想明白"
- 没提循环调用的实际问题——缺少生产经验
- 在"模型 vs 工程"上站一边——应该是互补

## 问题 7：模型选型与成本

**评分要点：**
- 能用数据做决策
- 会算总成本而非只看单价
- 了解路由（routing）架构

**满分回答示范：**

**小流量客服场景：**
选模型 A（准确率 95%，成本高）。客服场景的直接成本在整体业务中占比极低——一个客服坐席年薪 30 万，模型一天处理 1000 次对话的成本可能不到 $10。相比之下，一次 tool use 失败导致用户转人工或流失的代价远超模型成本的差异。这里应该优先准确率。

**大流量搜索场景：**
选模型 B（准确率 88%，成本低 10x）。搜索场景每 request 的利润极薄，且用户对偶尔的错误容忍度更高（大不了再搜一次）。100 万次搜索，模型 B 成本 $3000 vs 模型 A 成本 $30000。省下的 $27000 足够支付 12% 失败的处理成本（用户重试、降级到简单文本回复）还有大量盈余。

**总成本计算：**
设每百万次调用：
- 模型 A：$30000 + (5% × 失败处理成本 × 0.1 的概率需要人工 × $2/人工) = ~$31000
- 模型 B：$3000 + (12% × 30% 需要重试 × 额外成本 + 5% 转人工 × $2) = ~$9000

具体数字取决于业务模型，但 10x 的成本差距通常很难靠准确率弥补。

**Routing 架构（模型 A 只处理难 case）：**
- 设置一个"置信度阈值"。模型 B 输出 tool_call 时如果 logprob 低于阈值（表示模型不确定），自动升级到模型 A
- 或者用一个轻量级分类器判断问题难度，简单问题走模型 B，复杂问题走模型 A
- 工程上这叫"speculative routing"——便宜的先试，没把握再上贵的

**微调小模型的 Tool Use 数据：**
数据来源：
1. 用大模型（GPT-4）的正确 tool call 作为 teacher，蒸馏到小模型
2. 从生产日志中采集真实用户请求 + 正确的 tool call 路径，人工审核后加入训练集
3. 针对性构造对抗样本——把已知的失败 case 加入训练

**BFCL 和真实场景的 gap：**
BFCL（Berkeley Function Calling Leaderboard）测的是：给定一个 prompt + 工具定义，模型能否正确选择工具并填充参数。它的局限：
- **单轮为主**：真实场景多是 multi-turn
- **工具简单**：BFCL 的工具大多是 1-3 个参数的标准 API，真实场景的工具复杂得多（嵌套参数、条件必填）
- **没有延迟约束**：BFCL 不计时，但生产环境中 800ms vs 200ms 可能决定用户体验
- **没有对抗样本**：没有人故意 prompt injection 或给模糊输入
- **没有成本权重**：BFCL 只看准确率不看成本

**常见错误：**
- 只看单价不看总成本
- 不区分场景——一刀切选模型
- 不知道 BFCL 的局限——认为评测分数 = 生产表现
