# Tool Use / Function Calling — 考题答案

---

## 题 1 ⭐ — 选择题

**正确答案：B** — RAG 是 Tool Use 的一种特殊情况，其中"工具"是检索系统。

**解析：**
- A 错误：Tool Use 的范围远超信息获取，还包括执行操作（发邮件、改数据库等）。
- B 正确：RAG 的核心是调用一个检索工具（向量数据库/搜索引擎），将结果注入 prompt——这完全符合 Tool Use 的"暂停-调用-恢复"模式。
- C 错误：Tool Use 完全可以用于获取信息（如查天气、查股票）。
- D 错误：RAG 中也隐含了"工具"的概念（检索器本身就是一个隐式工具），且现代 RAG 越来越显式地用 Tool Use 实现。

## 题 2 ⭐ — 选择题

**正确答案：B** — 输出一个结构化的 tool call（包含工具名和参数），由框架拦截并执行。

**解析：**
- A 错误：如果是自然语言描述，外界没法可靠地解析出函数名和参数。
- B 正确：模型输出特殊 token（如 OpenAI 的 `tool_calls` 字段）或结构化 JSON，由推理框架拦截、解析、执行。
- C 错误：模型本身没有执行代码的能力，函数调用由外部框架完成。
- D 错误：不是模型"停止运行"，而是在一个推理序列中"暂停 → 注入结果 → 继续"。

## 题 3 ⭐⭐ — 选择题

**正确答案：B** — 同时查询北京、上海、广州三个城市的天气。

**解析：**
并行调用的前提是：多个调用**彼此独立**，不依赖对方的输出。
- A 错误：查询订单依赖用户 ID——必须先拿到 ID 再查订单，串行。
- B 正确：三个城市的天气互不依赖，完全可以并行。
- C 错误：插入依赖表已创建——串行。
- D 错误：提取链接依赖页面内容——串行。

## 题 4 ⭐⭐ — 简答题

**幻觉调用**是指模型调用了一个不存在的工具，或者给工具传了不合法/不存在的参数。

**例子：**
用户问"帮我查一下苹果公司的股价"，工具列表中有 `search_web(query)` 但没有 `get_stock_price(symbol)`。模型却输出了 `get_stock_price(symbol="AAPL")`——这个工具根本不存在。

**缓解策略：**

1. **Schema 约束**：推理框架在解析 tool call 时做严格校验——工具名必须在 tools 列表中，参数类型必须匹配 Schema。校验失败就直接告诉模型"这个工具不存在/参数不对，请重试"。

2. **在 system prompt 中强化边界**：明确写"只使用上面列出的工具。如果没有合适的工具，明确告诉用户你做不了，不要编造工具。"

3. **Few-shot 示例中展示拒绝场景**：在 system prompt 的 tool-use examples 中加入"没有合适工具时，模型应该回复'抱歉我无法完成这个请求'"的场景，强化模型的分辨能力。

4. **Fine-tuning 时加入 negative examples**：训练数据中不仅包含正确的 tool call，也包含"模型尝试调用不存在的工具 → 被拒绝 → 正确回复自己做不了"的样本。

## 题 5 ⭐⭐ — 简答题

**满分答案要点：**

**从模型决策角度：**
模型是根据 tool description 来决定"该不该用这个工具"以及"该用哪个工具"的。如果多个工具的描述相似但有细微差别，description 就是区分它们的关键信号。描述模糊 → 模型选错工具的概率大增。

**从信息匹配角度：**
用户意图和工具之间的语义匹配依赖 description。例如，用户说"帮我找个东西"，有 `search_files` 和 `search_emails` 两个工具。如果 description 写得足够具体（"搜索本地文件" vs "搜索邮箱中的邮件"），模型才能正确映射。

**不够好的后果：**
- **选择错误**：该用工具 A 却用了工具 B
- **不调用**：存在合适的工具但模型没有调用（以为自己做不了）
- **参数乱填**：description 没说清楚参数含义，模型猜测参数导致调用失败
- **过度调用**：简单问题也调用工具，增加延迟

**最佳实践：** description 应包含 (1) 工具的功能 (2) 什么时候用 (3) 什么时候不该用 (4) 参数的业务含义而非技术含义。

## 题 6 ⭐⭐⭐ — 场景题

**1. 调用顺序：**

用户说"我上周买的蓝色的包怎么还没到"，但没有提供订单号。模型应：

第一步：调用 `查询订单`，用用户身份 + 关键词（"蓝色"、"包"、"上周"）来定位具体订单。如果查询订单工具支持模糊搜索（按时间+商品描述），一步就能拿到订单。

第二步：拿到订单 ID 后，调用 `查询物流`(order_id)，获取物流详情。

第三步：根据物流状态回复用户。如果是正常的"运输中"，告知预计到达时间；如果是"异常"，提供处理方案（催件/补发/退款）。

如果第一步查不到订单（比如用户有多个蓝色包或时间对不上），应追问用户："您说的是哪个日期的订单？方便告诉我订单号吗？"

**2. 错误处理策略：**

收到 `{"status": "error", "message": "运单号不存在"}` 时：

- **不要直接暴露原始错误给用户**。将技术错误转化为用户友好的回复："抱歉，这个订单的物流信息暂时查询不到，可能是物流商系统延迟。我帮您转接人工客服确认一下？"
- **记录错误日志**：将原始错误 + 订单 ID + 时间写入日志，供排查。
- **提供 fallback 路径**：转人工客服或者建议用户稍后再查。
- **检查是否需要重试**：如果是网络抖动导致的临时错误（而非"不存在"这种业务错误），可以重试 1-2 次。

**3. 500 次调用分析：**

可能原因：
- **用户反复点击**：前端没有防抖/节流，用户多次点击触发多次请求
- **Agent 死循环**：模型不断重试同一个失败的调用
- **滥用/攻击**：脚本化的恶意请求

**防护措施：**
- **Rate limiting**：对每用户/每工具的调用频率设限（如每分钟 10 次）
- **Circuit breaker**：同一工具连续失败 N 次后自动熔断，返回 fallback 响应
- **Idempotency**：查询类操作天然幂等，但变更类（取消订单）需要幂等 key 防重复
- **Max turns**：限制单个对话轮次的 tool call 总次数（如最多 10 轮）
- **前端去抖**：按钮点击后 loading 状态，防止重复请求
