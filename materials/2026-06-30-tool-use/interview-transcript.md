# Tool Use / Function Calling — 面试实录

> 面试时间：2026-06-30  
> 面试风格：Google / ByteDance / Anthropic  
> 最终结果：7/7 通过 ✅

---

## 问题 1：LLM Tool Use 完整生命周期 (L1→L2)

**主问题：** 从用户发消息到模型回复，Tool Use 的完整生命周期？含异常分支。

**受访者回答：**
用户输入给模型 → 模型在 system prompt 中看到可用工具 → 模型输出结构化 tool call → 框架解析、schema 校验 → 有效则执行、无效则返回明确错误（区分函数名错误/参数错误等）→ 结果注入上下文。

**点评：**
- ✅ 基本流程正确，提到了 schema 校验和结构化错误反馈——很多候选人漏掉这点
- ⚠️ 没区分不同异常的处理策略（网络超时 vs 参数错误处理不同）
- ⚠️ 没提模型"决定不调用工具"的分支

---

**追问 1：** 工具执行超时怎么办？5s 和 30s 超时策略有何不同？

**受访者回答：**
返回明确超时信号和时间给模型，模型调用时可以传入超时参数判断是短期等待还是长时间轮询。

**点评：**
- ✅ 结构化错误回传思路对
- ❌ **纠正：超时不应由模型传入**——模型是决策层，超时阈值应在框架/工具定义层设定
- 正确做法：5s 超时 → 指数退避快速重试；30s → 切成异步轮询（返回 job_id 让模型隔几秒查）

---

**追问 2：** 模型怎么知道该在哪里停，输出 tool call？训练的还是 prompt 的？

**受访者回答：**
训练阶段，尤其是 post-training RLHF 阶段，给模型 tool call 的输入输出示例，正确格式给奖励。

**点评：**
- ✅ 训练决定，不是 prompt——核心判断对
- ⚠️ 缺了：特殊 token（如 `<tool_call>`）触发框架拦截的机制
- ⚠️ 缺了 2026 趋势：RLVR 替代 RLHF（NVIDIA Tool-N1 纯 RL 超越 SFT+RL）

---

**追问 3：** 一次输出 3 个 tool_call，框架串行还是并行？怎么判断？

**受访者回答：**
先用权限初筛——读权限可并行，写权限默认不并行，高危命令不行。

**点评：**
- ⚠️ **理解偏差**：并行的判断标准是**数据依赖**，不是读写权限。两个无依赖的写操作也可以并行
- ✅ 正确做法：同轮推理输出的 tool_call 默认无依赖，可并行；有依赖的由推理轮次自然分隔

---

## 问题 2：Schema 设计与语义理解 (L2)

**主问题：** 设计 `search_users` 和 `get_user`，description 怎么写让模型正确选择？

**受访者回答：**
search_users 查询符合条件的用户名称/ID 列表；get_user 根据 ID 查询完整信息。

**点评：**
- ✅ 功能描述准确
- ⚠️ 没回答核心：description 怎么写。需要双向互斥引导

---

**追问 1：** "帮我找一下张三"——用哪个？选错了改哪里？

**受访者回答：**
应该用 get_user("张三")。改 description 的输入输出和功能描述。

**点评：**
- ❌ **选错了**："找一下张三"没有确切 ID，应该是 search_users → 找到 ID → 再 get_user
- 正确 description 需包含"什么时候**不**用我"的负向引导

---

**追问 2：** search_users 返回 50 个结果怎么处理？

**受访者回答：**
看数据结构复杂度。数据量大时只返回 name + ID + 必要描述。可设阈值分段。

**点评：**
- ✅ 简洁模式 + 阈值分段策略
- ✅ 认识到目的是帮你挑用户、不是展示全量

---

## 问题 3：可靠性工程 (L3)

**主问题：** 设计容错体系，让用户看不到工具调用失败。

**受访者回答：**
在工具层上加渲染层，把错误转成用户友好的话术。

**点评：**
- ✅ 渲染层思路对
- ⚠️ 缺了重试、熔断、降级等前置环节

---

**追问 1：** 重试的退避策略？什么时候不该重试？

**受访者回答：**
指数退避 + 上限。超限后走熔断，有相同功能的替代走降级。

**点评：**
- ✅ 退避 + 熔断 + 降级思路
- ⚠️ 没区分可重试（网络超时/503/429）和不可重试（参数错误/权限不足/404）

---

**追问 2：** 3 次重试都失败，fallback 具体怎么做？

**受访者回答：**
兜底话术"当前服务不可用，请稍后再试"。查询类可给缓存或示例数据。

**点评：**
- ✅ 通用兜底 + 缓存降级——生产常用策略

---

## 问题 4：Multi-Turn 上下文管理 (L3→L4)

**主问题：** 15 轮对话 6 次工具调用，上下文快满了怎么办？

**受访者回答：**
滑动窗口保留最近消息 + LLM 总结非近期内容。远期的工具输出替换为摘要（执行成功、返回了多少数据等）。

**点评：**
- ✅ 滑动窗口 + 摘要策略

---

**追问：** 怎么判断某次工具调用结果后续还需要？

**受访者回答：**
不太清楚。

**点评：**
- ⚠️ **知识盲区**：正确做法是构建依赖图——被后续轮次引用过的保留，孤立节点优先压缩

---

## 问题 5：安全与权限 (L3)

**主问题：** 多租户 SaaS，如何保证 Agent 不越权？

**受访者回答：**
工具有自己的权限，用户权限集传递给 Agent，决定工具对 Agent 可见性。

**点评：**
- ✅ 权限层级思路对
- ⚠️ 没提 Tool call 携带的是**用户身份**（不是 Agent 身份）

---

**追问 1：** delete_user 没设 user_id 参数（隐式当前用户）的风险？

**受访者回答：**
其他模块调用时安全校验可能失效。

**点评：**
- ⚠️ 接近但不够具体：核心风险是 **prompt injection**——只要诱导调用 delete_user，当前用户就被删
- ✅ 安全设计：变更类操作强制传 user_id，后端校验归属

---

**追问 2：** Human-in-the-Loop 怎么设计？

**受访者回答：**
工具危险等级分层，高危险等级的调用必须人工确认。

**点评：**
- ✅ 四级分层：L0 自动 → L1 软确认 → L2 硬确认 → L3 人工审核

---

## 问题 6：Agent 自主性与编排 (L3→L4)

**主问题：** LLM 工具编排的最大局限？

**受访者回答：**
长程能力——哪些内容需要沉淀为偏好、配置了什么级别的环境，新开会话就丢失。

**点评：**
- ⚠️ 偏 memory/state 方向。编排的真正局限是**局部最优无全局规划**——不会回溯、容易走入死胡同

---

**追问：** Agent 陷入循环怎么打断？

**受访者回答：**
检测连续 N 次重复入参，检测循环调用模式。

**点评：**
- ✅ 重复检测 + 循环检测
- 补：N 轮无实质进展检测（即使参数变了但没获取新信息也打断）

---

## 问题 7：模型选型与成本 (L4)

**主问题：** A (95%/800ms/$0.03) vs B (88%/200ms/$0.003)，客服和搜索怎么选？

**受访者回答：**
客服选 A（容错空间小），搜索选 B（节约成本，用户可重试）。

**点评：**
- ✅ 场景判断正确

---

**追问：** 模型 A 只处理 B 搞不定的 case 怎么工程化？

**受访者回答：**
主 Agent 路由——识别场景 → 客服转 A，搜索转 B → B 搞不定交给 A。

**点评：**
- ✅ 路由架构思路正确
- 补充：speculative routing——看 B 输出 tool_call 的 logprob，低于阈值自动升级到 A，比场景路由更精细

---

## 总体评价

### 做的好的
- 基础流程扎实：生命周期、Schema 设计、错误处理
- 工程意识有亮点：容错分层、危险等级、路由架构
- 能区分训练 vs prompt 驱动

### 需要加强的
- **并行判断标准**：数据依赖，不是读写权限
- **RLVR 趋势**：2026 年 RLHF → 纯 rule-based RL
- **依赖图**：用于上下文管理和判断工具结果是否保留
- **Speculative routing**：基于置信度的模型路由
- **Prompt injection**：工具调用场景下的攻击面和防护
