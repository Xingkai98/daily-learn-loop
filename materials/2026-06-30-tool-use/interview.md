# Tool Use / Function Calling — 模拟面试

面试风格：对标 Google/ByteDance/Anthropic — 从基础原理到工程实践，逐层追问。

---

## 问题 1：基础深挖 (L1→L2)

**主问题：** 请解释 LLM Tool Use 的完整生命周期——从用户发消息到模型回复，中间发生了什么？不要只讲 happy path，把可能的异常分支也讲进去。

**追问链：**
- 你提到模型输出 tool call 后框架去执行——如果执行超时了怎么办？5 秒超时和 30 秒超时的处理策略有什么不同？
- 模型的 token 生成怎么知道该在哪里停下来进入 tool call？是训练的还是靠 prompt？
- 如果模型同时决定调用 3 个工具，推理引擎怎么处理？是串行还是并行？怎么判断哪些可以并行？

---

## 问题 2：Schema 设计与语义理解 (L2)

**主问题：** 你有一个用户系统，需要设计 `search_users` 和 `get_user` 两个工具。它们的 description 怎么写才能让模型在"模糊搜索"和"精确查询"之间做出正确选择？

**追问链：**
- 如果用户说"帮我找一下张三"，模型该用哪个工具？如果你的设计让模型选错了，改哪里？
- description 里写"用于搜索用户"就够了，还是需要更多信息？具体需要什么？
- 如果 search_users 返回了 50 个结果，下一步怎么处理？直接把 50 个 JSON 塞给模型吗？
- 你如何衡量一个 tool schema 设计得好不好？有没有量化指标？

---

## 问题 3：可靠性工程 (L3)

**主问题：** Tool Use 在生产环境中的失败率通常在 3-10%。请设计一套容错体系，目标是让用户看不到任何工具调用失败。

**追问链：**
- 你提到重试——重试的退避策略是什么？什么时候不该重试？
- 如果重试 3 次都失败了，fallback 策略是什么？直接告诉用户"出错了"还是走降级路径？
- 工具返回的数据格式和你预期的 Schema 不一致——字段名变了、类型变了，怎么处理？
- 你如何监控工具调用的质量？需要哪些 metrics？定义一下"调用成功率"。

---

## 问题 4：Multi-Turn & 状态管理 (L3→L4)

**主问题：** 用户和 Agent 进行了 15 轮对话，其中调用了 6 次工具。到了第 15 轮，上下文窗口快满了——前面工具调用的结果占了很多 token。你会怎么设计这个系统的上下文管理？

**追问链：**
- 工具返回的原始 JSON 和模型重新组织的回复——两者都要保留在上下文中吗？为什么？
- 怎么判断某次工具调用的结果在后续对话中是否还需要？
- 如果你设计一个工具调用结果的"摘要器"（把长结果压缩成短摘要），要怎么做？有哪些风险？
- 这和 KV Cache 管理有什么关系？

---

## 问题 5：安全与权限 (L3)

**主问题：** 在一个多租户 SaaS 系统中，Agent 可以调用 `delete_user`、`query_order` 这类敏感工具。如何保证 Agent 不会越权操作？

**追问链：**
- Tool call 携带的是用户身份还是 Agent 身份？权限校验在哪一层做？
- 如果 `delete_user` 的 tool schema 中没有提供 user_id 参数（隐式用当前用户），有什么安全风险？如何设计才安全？
- 大模型可能被 prompt injection 诱导调用不该调的工具——怎么防护？
- 你能设计一个"人工确认"（Human-in-the-Loop）的机制吗？哪些操作需要确认，哪些不需要？

---

## 问题 6：Agent 自主性 (L3→L4)

**主问题：** 有些场景下，Agent 需要"自己决定"调用什么工具、按什么顺序——这就是工具编排（Orchestration）。当前 LLM 的工具编排能力有多大局限？举个实际翻车的例子。

**追问链：**
- ReAct 模式和纯 Tool Use 有什么区别？ReAct 更适合什么场景？
- 如果 Agent 落入了"循环调用"——不断用不同的参数调用同一个工具但永远得不到满意结果——你能从工程上怎么打断这个循环？
- 你怎么区分"Agent 确实需要多步推理"和"Agent 在瞎转"？能不能在 3 轮工具调用内判断？
- 长期来看，你觉得工具编排应该靠模型能力提升还是靠工程框架约束？

---

## 问题 7：模型选型与成本 (L4)

**主问题：** 你需要在两个模型之间做选择：模型 A Tool Use 准确率 95%，推理延迟 800ms，成本 $0.03/1K tokens。模型 B 准确率 88%，延迟 200ms，成本 $0.003/1K tokens。给小流量客服场景和大流量搜索场景分别怎么选？为什么？

**追问链：**
- 你说大流量场景选便宜的——但 88% 的准确率意味着 12% 的调用会失败，你要花多少钱处理这些失败？把总成本算一下。
- 能不能让模型 A 只处理模型 B 搞不定的 case？这怎么工程化？
- 你提到用微调的小模型做 Tool Use——微调需要什么数据？数据从哪来？
- Tool Use 的评测到底怎么做的？BFCL 和真实场景的 gap 是什么？
